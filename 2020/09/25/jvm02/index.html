运行时数据区

红色区域是一个进程一份，共享的：

  • 堆
  • 堆外内存(方法区)（永久代，元空间，代码缓存）

灰色区域是一个线程一份，私有的

  • 程序计数器
  • 栈
  • 本地栈

线程

线程是一个程序里的运行单元，JVM允许一个应用有多个线程并行的执行

  • hotspot虚拟机中，每个线程都与操作系统的本地线程直接映射，当java线程准备好以后，会创建一个操作系统的本地线程。java线程终止以后，本地线程也会回收。
  • 操作系统负责把线程调度到可用的cpu上，去调用java线程中的run()方法
  • 当最后一个普通线程结束的时候（main方法）虚拟机也会被关闭

程序计数器(PC寄存器)

介绍

不是cpu寄存器，是jvm中对物理pc寄存器的一种抽象。

pc寄存器用来存储指向下一条指令的地址，也即将要执行的指令代码。由执行引擎读取下一条指令

他是线程私有的，生命周期与线程的生命周期保持一致。

任何时间一个线程都只有一个方法在执行，也就是所谓 的当前方法。程序计数器会存储当前线程正在执行的java方法的jvm指令地址。或者如果是在执行native方法，则是未指定值(undefined)

程序计数器是唯一一个在java虚拟机中没有规定任何OutOfMemoryError情况的区域

举例说明

public static void main(String[] args) {
    int i = 10;
    int j = 20;
    int k = i+j;
    String s = "abc";
    System.out.println(i);
    System.out.println(k);
}

常见问题

使用PC寄存器存储字节码指令地址有什么用呢？

cpu需要不停的切换各个线程，这时候切换回来后，就得知道接着从哪里开始继续执行。

为什么使用pc寄存器记录当前线程都执行地址呢？

JVM的字节码解释器就需要通过改变PC寄存器的值来明确下一条应该执行什么样的字节码指令

虚拟机栈

概述

虚拟机栈跨平台容易，性能要比寄存器的低。

  • 每个线程在创建的时候都会创建一个虚拟机栈，其内部保存一个个的栈帧，对应着一次次的java方法调用
  • 是线程私有的
  • 生命周期和线程一致
  • 主管java程序的运行，保存方法的局部变量（8种基本数据类型，对象的引用地址），部分结果，并参与方法的调用和返回

特点：

  • 不存在垃圾回收问题
  • 每个方法的执行都伴随着压栈
  • 执行结束后会有出栈工作

栈中出现的异常

java虚拟机规范java栈的大小是动态的或者固定不变的

  • 采用固定大小的虚拟机栈，在线程创建的时候，要考虑自己的局部变量表什么的是否超过了java虚拟机栈允许的最大容量，此时会抛出一个StackOverflowError
  • 采用动态扩展的虚拟机栈，在尝试扩展的时候没办法申请到足够的内存，或者在创建的时候就没有足够的内存去创建虚拟机栈，那么会抛出一个OutOfMemoryError
public class StackErrorTest {
    public static void main(String[] args) {
        main(args);
    }
}

以上代码会抛出异常：

Exception in thread "main" java.lang.StackOverflowError
	at com.study.StackErrorTest.main(StackErrorTest.java:10)

我们可以通过-Xss 具体内存来控制栈内存的大小

栈的存储单位

栈里的数据都是以栈帧(Stack Frame)的格式存在的。

在线程上正在执行的每个方法都各自对应一个栈帧

栈帧是一个内存区块，数据集

  • 不同的线程之间所包含的栈帧是不能相互调用的

  • 如果当前方法调用了其他方法，方法返回的时候会把结果给前一个栈帧，自己被弹出栈

  • 无论是正常的return返回，还是抛出异常结束（没有处理掉），都会导致栈帧被弹出

    如图中所示，当方法4出现异常，自身没有处理，栈就会弹出方法4,把异常返回给方法3，看看方法3是否处理，如果还没有处理，则栈弹出方法3并继续返回，直到最后方法1。如果方法1没有处理，那么虚拟机就会挂掉。

栈桢的内部结构

  • 局部变量表
  • 操作数栈
  • 动态链接（或指向运行时常量池的方法引用）
  • 方法返回地址（或方法正常退出或者异常退出的定义）
  • 一些附加信息

局部变量表

被称为局部变量数组或者本地变量表

定义为一个数字数组，主要用于存储方法参数和自定义在方法体内的局部变量,这些数据类型包括各类基本数据类型，对象引用，以及returnAddress类型

  • 建立在线程的栈上，是线程私有数据，不存在线程安全问题
  • 局部变量表所需的容量大小事在编译的时候就确定下来的，并且保存在方法的Code属性的maximun local variables数据项中。运行期间不会改变局部变量表的大小
  • 局部变量表的变量只在当前方法调用中有效，调用结束后，随着方法栈帧的销毁，局部变量表也会随之销毁

slot

slot 是局部变量表中最基本的存储单元

在局部变量表中存放编译期可知道各种数据类型，引用类型(reference)，returnAddress类型的变量

32位以内的类型只占用一个slot，64位的站俩

一个问题：

 public static void test123(){
    // 这里的this会导致编译失败，因为静态方法没有this存在局部变量表中
    System.out.println(this);
}

// 相反的，在我们的实例创建的方法，或者构造方法中，this会存放在index为0的slot中
public test(){
    sout(this)//是可以的
}

slot重复利用问题

栈帧中的局部变量表中的slot是可以重用的，如果一个局部变量过了他的作用域，在其后声明的局部变量就可能会使用他的槽位，从而节省空间

/**
因为b的生命域就在{}中，出了{}他就被销毁了。但是在之前虚拟机栈已经为他开辟了空间，销毁后让c使用他的位置
*/
public void test(){
    int a = 10;
    {
        int b = 0;
        b = a +1;
    }

    int c = a + a;
}

操作数栈

特点：后进先出

在方法执行的过程当中，根据字节码指令，往栈中写入数据或者提取数据，（入栈/出栈）

- 比如执行复制，交换，求和等操作

如果被调用的方法带有返回值，返回值会被压入当前栈帧的操作数栈中，并更新pc寄存器中下一条需要执行的字节码指令。

  • 虚拟机在操作数栈中存储数据的方式和在局部变量区中是一样的：如int、long、float、double、reference和returnType的存储。对于byte、short以及char类型的值在压入到操作数栈之前，也会被转换为int。

虚拟机把操作数栈作为它的工作区——大多数指令都要从这里弹出数据，执行运算，然后把结果压回操作数栈。

begin  
iload_0    // push the int in local variable 0 onto the stack  
iload_1    // push the int in local variable 1 onto the stack  
iadd       // pop two ints, add them, push result  
istore_2   // pop int, store into local variable 2  
end

在上边的例子中，iload_0和iload_1把局部变量表中索引为0和1的整数压入操作数栈

调用iadd指令把两个数从操作数栈中弹出并相加，把结果压入操作数栈

istore_2从操作数栈中弹出结果，把它存储到局部变量表中索引为2的位置。

栈顶缓存技术

基于栈式架构的虚拟机使用的零地址指令更加紧凑，但是完成一个操作需要不停的入栈出栈，，经常对内存进行读写，效率会被降低。

所以开发者将栈顶元素全部缓存在物理cpu寄存器中，以此降低对内存的读写次数，提升执行引擎的执行效率

动态链接

在某些书上 动态链接，方法返回地址，附加信息会被称为帧数据区

  • 动态链接主要就是指向运行时常量池的方法引用
  • 每一个栈帧内存都包含一个指向运行时常量池中该栈帧所属方法的引用，包含这个引用的目的就是为了支持当前方法的代码能够实现动态链接(Dynamic Linking)。比如invokedynamic 指令
  • 在Java源文件被编译到字节码文件中时，所有的变量和方法引用都作为符号引用保存在class文件的常量池里。比如，描述一个方法调用其他方法时，就是通过常量池中指向方法的符号引用来表示的，那么动态链接的作用就是为了将这些符号引用转换为调用方法的直接引用

方法的调用：解析与分派

将符号引用转换成调用方法的直接引用

如果转换的过程是在编译期间确定下来，并且运行期间保持不变，那么这就是静态链接

如果被调用的方法在编译期无法被确定下来，那么只能在程序运行期间进行转换，被称为动态链接

对于方法来说：

接口的方法调用在编译期间是不确定的，所以是晚期绑定

而方法中如果调用了构造器，是可以在编译期间确定的，是早期绑定

虚方法和非虚方法：

  • 静态方法，私有方法，final方法，构造器，父类方法都是非虚方法
  • 方法在编译器就确定了具体的调用，并且在运行时不可变，就是非虚方法

JVM提供了如下方法调用指令：

  1. invokestatic 调用静态方法
  2. invokespecial 调用私有实例方法、构造器，以及使用super关键字调用父类的实例方法或构造器，和所有实现接口的默认方法。
  3. invokevirtual 调用虚方法即非私有的实例方法。
  4. invokeinterface 调用接口方法，在运行时再确定一个实现此接口的对象。
  5. invokedynamic 调用动态方法，在运行时动态解析出调用点限定符所引用的方法之后，调用该方法。

方法重写的本质

虚方法表

在面向对象的编程中，会很频繁的使用到动态分派，如果在每次动态分派的过程中都要重新在类的方法元数据中搜索合适的目标的话就会影响到执行效率。

为了提高性能，jvm在类的方法区建立一个虚方法表。使用索引来代替查找。

每个类都有一个虚方法表，用来存放各个方法的实际入口

虚方法表在什么时候会被创建？？

虚方法表会在类加载到链接阶段被创建并开始初始化，类的变量初始值准备完成后，JVM会把该类的方法表也初始化完毕。

方法返回地址

这一块区域是用来存储pc寄存器中的值，目的是为了给执行引擎让他继续做操作

一个方法如果是正常执行完成时pc寄存器的值作为返回地址，即调用该方法的指令的下一条指令的地址。如果是通过异常退出的，返回地址要通过异常表来确定，栈帧中一般不会保存此信息

本地方法栈

native method就是java调用非java代码的接口

本地接口的初衷就是融合不同的编程语言为java所用

本地方法栈是用于管理本地方法的调用

  • 本地方法是线程私有的
  • 允许被实现成固定或者是可动态扩展的内存大小
    • 如果线程请求分配的栈容量超过本地方法栈允许的最大容量报错栈溢出
    • 如果本地方法栈可以动态扩展，在尝试扩展的时候无法申请到足够的内存就会内存溢出
  • 本地方法是c实现的
  • 在本地方法栈中登记native方法，在执行引擎执行时候加载本地方法库

  • 当某个线程调用一个本地方法时，他会进入一个全新的并且不受虚拟机限制的世界
    • 本地方法可以通过本地方法接口来访问虚拟机内部的运行时数据区
    • 甚至可以直接使用本地处理器中的寄存器
    • 直接从本地的堆中分配任意数量的内存
  • 并不是所有的JVM都支持本地方法，因为Java虚拟机规范没有明确要求本地方法站的使用语言，具体实现方式，数据结构。
  • 在 Hotspot虚拟机中，直接将本地方法栈和虚拟机合二为一
